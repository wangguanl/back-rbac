## 1. 登出不完整

### 现状

`Header.vue` 退出时只做了本地清理，未走完整登出流程：

```52:61:frontend/src/components/layout/Header.vue
async function handleCommand(command: string) {
  if (command === 'logout') {
    await ElMessageBox.confirm('确定要退出登录吗？', '提示', {
      confirmButtonText: '确定',
      cancelButtonText: '取消',
      type: 'warning'
    })
    userStore.resetState()
    router.push('/login')
  }
}
```

问题点：

- 未调用 `logoutApi()`，后端 Redis 中的 token 不会被清除
- 未调用 `permissionStore.resetRoutes()`，动态路由仍留在 Vue Router 实例中
- 项目已有 `useAuth().logout()`，但 Header 未使用，且 `useAuth` 本身也不完整

### 影响

- 换账号登录时，上一用户的动态路由可能残留，导致菜单/路由错乱
- 后端认为会话仍有效（见缺口 3），存在安全风险

### 解决方案

**统一登出入口**：所有退出逻辑收敛到 `useAuth().logout()`，Header 只负责触发。

#### 1.1 扩展 `permission` store — 增加路由重置

```typescript
// frontend/src/stores/permission.ts
import router from '@/router'

const addedRouteNames = ref<string[]>([])

async function loadRoutes() {
  // ... 现有逻辑 ...
  const names = routes.value.map(r => r.name).filter(Boolean) as string[]
  addedRouteNames.value = names
  return routes.value
}

function resetRoutes() {
  addedRouteNames.value.forEach(name => {
    if (router.hasRoute(name)) router.removeRoute(name)
  })
  addedRouteNames.value = []
  routes.value = []
  menuList.value = []
}
```

> 注意：当前 `generateRoutes` 未设置 `name`，需在生成路由时补充 `name: menu.name`（或与后端约定唯一标识），否则 `removeRoute` 无法精准移除。备选方案：记录 `path` 后用 `router.getRoutes()` 过滤移除。

#### 1.2 完善 `useAuth().logout()`

```typescript
// frontend/src/composables/useAuth.ts
import { logoutApi } from '@/api/auth'

async function logout(options?: { confirm?: boolean }) {
  if (options?.confirm !== false) {
    try {
      await ElMessageBox.confirm('确定要退出登录吗？', '提示', { /* ... */ })
    } catch {
      return
    }
  }

  try {
    await logoutApi()          // 通知后端清除 Redis token
  } catch {
    // 网络失败也继续本地清理，避免用户卡在已登出态
  }

  permissionStore.resetRoutes()
  userStore.resetState()
  router.replace('/login')
}
```

#### 1.3 Header 改为调用 composable

```typescript
// frontend/src/components/layout/Header.vue
import { useAuth } from '@/composables/useAuth'

const { logout } = useAuth()

async function handleCommand(command: string) {
  if (command === 'logout') await logout()
}
```

#### 1.4 401 拦截器同步清理

`request.ts` 在 401 时除 `removeToken()` 外，应同样调用 `userStore.resetState()` + `permissionStore.resetRoutes()`，避免 token 过期后路由残留。

### 验收标准

- [ ] 点击退出 → 请求 `POST /auth/logout` 成功
- [ ] 换账号登录后，侧边栏与路由仅显示新用户权限
- [ ] 401 响应后刷新页面，不会短暂看到上一用户菜单

## 2. 双 token 存储

### 现状

Token 同时存在于两处：

| 存储位置 | 写入时机 | 读取时机 |
|---------|---------|---------|
| Pinia persist `localStorage['user']` | `login()` 赋值 `token.value` | 路由守卫 `userStore.token` |
| `localStorage['rbac_token']` | `setToken()` | Axios 拦截器 `getToken()` |

```12:16:frontend/src/stores/user.ts
  async function login(params: { username: string; password: string }) {
    const res = await loginApi(params)
    token.value = res.data.token
    setToken(res.data.token)   // 重复写入
    return res.data
  }
```

```26:31:frontend/src/stores/user.ts
  function resetState() {
    token.value = ''
    // ...
    removeToken()              // 只清 rbac_token，Pinia persist 也会清 token 字段
  }
```

理论上若只改一处（如手动改 localStorage、或 persist 插件异常），两处会不同步，导致「守卫认为已登录但请求不带 token」或相反。

### 解决方案

**单一数据源（推荐）**：只保留 Pinia persist，移除 `rbac_token`。

#### 2.1 改造请求拦截器

```typescript
// frontend/src/utils/request.ts
import { useUserStore } from '@/stores/user'

service.interceptors.request.use(config => {
  const token = useUserStore().token
  if (token) config.headers.Authorization = `Bearer ${token}`
  return config
})
```

#### 2.2 简化 user store

```typescript
async function login(params) {
  const res = await loginApi(params)
  token.value = res.data.token
  // 删除 setToken()
  return res.data
}

function resetState() {
  token.value = ''
  userInfo.value = null
  permissions.value = []
  // 删除 removeToken()，persist 插件会自动同步
}
```

#### 2.3 清理废弃文件

- 删除或标记废弃 `utils/storage.ts`、`utils/auth.ts`（若仅 re-export token 函数）
- 全局搜索 `getToken` / `setToken` / `removeToken` 并替换

#### 2.4 迁移兼容（可选）

首次加载时若检测到旧 key `rbac_token` 且 persist 中无 token，可一次性迁移后删除旧 key：

```typescript
const legacy = localStorage.getItem('rbac_token')
if (legacy && !useUserStore().token) {
  useUserStore().token = legacy
  localStorage.removeItem('rbac_token')
}
```

### 备选方案

若希望 HTTP 层与 Store 解耦，可反过来：**只保留 `rbac_token`，Pinia persist 的 `pick` 去掉 `token`**，守卫通过 `getToken()` 判断登录态。两种方案选其一，不要并存。

### 验收标准

- [ ] `localStorage` 中仅有一处 token（或 persist 包内一处）
- [ ] 登录、刷新、登出、401 全流程 token 读写一致

## 3. Redis token 未参与校验

### 现状

登录时将 token 写入 Redis：

```28:29:backend/src/modules/auth/auth.service.ts
    // 缓存 token 到 Redis (用于登出)
    await redis.set(`token:${user.id}`, token, 'EX', 7200)
```

登出时删除：

```47:49:backend/src/modules/auth/auth.service.ts
  async logout(userId: number) {
    await redis.del(`token:${userId}`)
  }
```

但 `authMiddleware` 只校验 JWT 签名与用户状态，**不比对 Redis**：

```15:20:backend/src/middleware/auth.middleware.ts
export async function authMiddleware(req: Request, res: Response, next: NextFunction) {
  const token = req.headers.authorization?.replace('Bearer ', '')
  if (!token) throw new UnauthorizedException('未登录')
  const payload = verifyToken(token)
  // 未检查 redis token:${userId}
```

因此登出后 JWT 在过期前（默认 2h）仍然有效。

### 解决方案

在 `authMiddleware` 中增加 Redis 会话校验，与 JWT 过期时间对齐。

#### 3.1 中间件改造

```typescript
// backend/src/middleware/auth.middleware.ts
import redis from '@/utils/redis'

export async function authMiddleware(req: Request, res: Response, next: NextFunction) {
  const token = req.headers.authorization?.replace('Bearer ', '')
  if (!token) throw new UnauthorizedException('未登录')

  const payload = verifyToken(token)

  // Redis 会话校验：登出 / 踢人 / 换端登录后立即失效
  const cachedToken = await redis.get(`token:${payload.userId}`)
  if (!cachedToken || cachedToken !== token) {
    throw new UnauthorizedException('登录已失效，请重新登录')
  }

  // ... 原有用户查询与权限聚合 ...
}
```

#### 3.2 TTL 与 JWT 对齐

当前 Redis TTL 硬编码 `7200`（2h），JWT 默认也是 `2h`，应统一从 config 读取：

```typescript
// auth.service.ts
import { config } from '@/config'

const ttlSeconds = parseJwtExpiresIn(config.jwt.expiresIn) // 如 '2h' → 7200
await redis.set(`token:${userId}`, token, 'EX', ttlSeconds)
```

#### 3.3 Redis 不可用时的策略

| 策略 | 适用场景 |
|------|---------|
| **fail-closed**（推荐生产） | Redis 异常 → 返回 503/401，安全优先 |
| **fail-open**（仅开发） | Redis 异常 → 跳过校验，方便本地无 Redis 调试 |

可通过环境变量 `REDIS_SESSION_CHECK=true|false` 控制。

#### 3.4 单点登录 / 踢人（可选增强）

当前 `token:${userId}` 为单 key，新登录会覆盖旧 token，天然实现「后登录踢掉先登录」。若需多端共存，改为 `token:${userId}:${deviceId}` 集合结构。

### 验收标准

- [ ] 登录 → 登出 → 用旧 token 请求任意受保护接口 → 401
- [ ] 登录 → 再次登录（新 token）→ 旧 token 请求 → 401
- [ ] 正常请求不受影响

## 4. 日志模块 HTTP 未接

### 现状

| 层级 | 状态 |
|------|------|
| 数据库表 `sys_log` / `sys_login_log` | ✅ 已建 |
| `log.service.ts` `record()` | ✅ 可写操作日志 |
| `auth.service` `recordLoginLog()` | ✅ 登录时写入 |
| HTTP 查询接口 | ❌ 未注册 |
| 前端日志管理页 | ❌ 未实现 |

`app.ts` 仅注册了 auth / users / roles / menus，无 `/api/logs`。

### 解决方案

按 [RBAC后端方案.md](./RBAC后端方案.md) 5.5 节补齐查询能力。

#### 4.1 后端 API

```
GET  /api/logs/login          登录日志分页列表    权限: log:list
GET  /api/logs/operation      操作日志分页列表    权限: log:list
GET  /api/logs/operation/:id  操作日志详情        权限: log:query
```

**目录结构**：

```
backend/src/modules/log/
├── log.controller.ts
├── log.route.ts
├── log.service.ts      # 已有 record()，补充 findLoginLogs / findOperationLogs / findById
└── log.dto.ts          # 分页查询参数
```

**log.service.ts 扩展示例**：

```typescript
async findLoginLogs(query: { page: number; pageSize: number; username?: string; status?: number }) {
  const { page, pageSize, username, status } = query
  const where = {
    ...(username && { username: { contains: username } }),
    ...(status !== undefined && { status })
  }
  const [list, total] = await Promise.all([
    prisma.sysLoginLog.findMany({
      where,
      skip: (page - 1) * pageSize,
      take: pageSize,
      orderBy: { createTime: 'desc' }
    }),
    prisma.sysLoginLog.count({ where })
  ])
  return { list, total, page, pageSize }
}
```

**注册路由**：

```typescript
// backend/src/app.ts
import logRoutes from '@/modules/log/log.route'
app.use('/api/logs', logRoutes)
```

#### 4.2 操作日志自动记录（可选，P1+）

`log.service.record()` 已存在但未调用。两种接入方式：

| 方式 | 说明 |
|------|------|
| 日志中间件 | 在 `app.ts` 全局挂载，记录 method / url / 耗时 / 状态 |
| 装饰器 / 手动 | 在 user/role/menu 的增删改 controller 中显式调用 |

建议先用**中间件**覆盖写操作（POST/PUT/DELETE），参数做脱敏（password、token 字段）。

#### 4.3 前端（可选，P2）

若需管理界面：

```
frontend/src/views/system/log/
├── index.vue              # Tab：登录日志 / 操作日志
└── components/
    └── LogDetailDrawer.vue
```

- API：`frontend/src/api/log.ts`
- 菜单 seed：增加「日志管理」菜单项，permission 为 `log:list`
- 复用现有表格 + 分页 composable 模式（参考 user/role 页）

#### 4.4 Seed 权限

确保 admin 角色分配 `log:list`、`log:query` 菜单权限，否则接口会被 `requirePermission` 拦截。

### 验收标准

- [ ] `GET /api/logs/login?page=1&pageSize=10` 返回分页数据
- [ ] `GET /api/logs/operation` 同上
- [ ] 无 `log:list` 权限的用户 → 403
- [ ] （可选）前端日志页可查询、查看详情

## 5. 文档示例代码过时

### 现状

[RBAC前端方案.md](./RBAC前端方案.md) 中的示例与真实代码差异较大，详见 [RBAC前端方案 对比差异.md](./RBAC前端方案%20对比差异.md)：

|---------|---------|
| `sessionStorage` 存 token | `localStorage` + Pinia persist |
| `asyncRoutes` + permissions 过滤 | 后端 `/menus/routes` + `import.meta.glob` |
